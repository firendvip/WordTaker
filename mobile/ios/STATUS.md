# WordTaker iOS — STATUS

This is the iPhone (iOS) version of WordTaker: **speak Chinese → transcribe
on-device → polish via the self-hosted relay (DeepSeek, key server-side) →
insert / copy text.**

## Honest build status

- **No `.ipa` or compiled build exists.** This Mac has only the Xcode
  command-line tools (`/Library/Developer/CommandLineTools`), **not full Xcode**.
  `xcodebuild` is therefore unavailable, and the iOS Simulator cannot run here.
  I did **not** fabricate a build.
- **The `.xcodeproj` was generated successfully** with XcodeGen
  (`xcodegen generate` → `WordTaker.xcodeproj`). XcodeGen 2.45.4 was installed
  via Homebrew on this machine.
- All Swift sources in the `Shared/` module pass `swift -frontend -parse`
  (clean). The SwiftUI/UIKit files (`ContentView`, `SpeechRecognizer`,
  `KeyboardViewController`) cannot be fully type-checked locally because the iOS
  SDK is absent; they have no pure-syntax errors. Final compilation must happen
  in full Xcode or via the CI workflow.

### App flow

1. Tap the record button → request mic + speech permissions.
2. Capture audio and transcribe live with `SFSpeechRecognizer` (locale `zh-CN`),
   preferring on-device recognition when supported. This replaces the desktop's
   local FunASR, which cannot run on iOS.
3. On stop, POST the recognized text to the relay for polish.
4. Show the polished result with a **Copy** button and a **polish on/off toggle**
   (raw vs polished). On relay failure it **falls back to the raw transcript**
   and shows a soft notice — the user never loses their text.
5. The final text is also written to the shared App Group store so the keyboard
   can insert it.

### Relay call

- `POST https://1311262545-3ihll1gdlf.ap-guangzhou.tencentscf.com`
- Body: `{"text":"...","mode":"copywriting"}`
- Headers: `Content-Type: application/json`,
  `X-App-Token: <app token>`, `X-Device-Id: <persistent random id>`
- Response: `{"success":true,"text":"polished"}` (non-streaming).
- `URLSession` async/await, 10s timeout, full error handling.
- **No DeepSeek key anywhere in the client** — only the relay URL and the
  `X-App-Token` (which the relay can rotate/revoke).

## Apple keyboard-microphone limitation (HONEST)

**Live microphone capture inside an iOS keyboard extension is not reliably
supported and may be rejected by App Review.**

- Keyboard extensions are sandboxed. Networking and reading the App Group
  container both require the user to enable **"Allow Full Access"**
  (`RequestsOpenAccess = true` in the keyboard's Info.plist). Many users never
  enable it.
- Even with Full Access, Apple restricts/throttles audio capture (`AVAudioEngine`
  / `AVAudioSession`) inside keyboard extensions, and such usage is commonly
  flagged in review. Treating a keyboard as a live recorder is fragile.

**What this project does instead (the supported path):**

1. Microphone capture + transcription + polish happen in the **host app**, where
   they are fully supported.
2. The result is written to the **App Group** (`SharedDraftStore`).
3. The **keyboard** reads the latest shared draft and inserts it with
   `textDocumentProxy.insertText(...)`.

The keyboard's mic button therefore directs the user to open the host app to
record, then return and tap **"插入最新文本"** to insert. This is honest about
the platform constraint rather than pretending in-keyboard mic works.

> If you still want to attempt in-keyboard capture, do it only behind a Full
> Access check and expect possible App Review pushback.

## How to open & build in Xcode

Prerequisites: a Mac with **full Xcode** (not just command-line tools) and an
Apple Developer account for device builds.

1. Open the project:
   ```bash
   cd "mobile/ios"
   open WordTaker.xcodeproj
   ```
   If `WordTaker.xcodeproj` is missing or you change `project.yml`, regenerate:
   ```bash
   brew install xcodegen   # once
   xcodegen generate
   ```
2. **Simulator (no signing):** select the `WordTaker` scheme + an iPhone
   simulator and press Run. Speech/mic permission prompts appear on first record.
   (Note: the Simulator's mic uses the Mac's mic.)
3. **Device build (signing required):**
   - Select the `WordTaker` target → **Signing & Capabilities** → choose your
     Team. Repeat for the `WordTakerKeyboard` target.
   - Set a real `DEVELOPMENT_TEAM` (in Xcode UI, or in `project.yml` then
     `xcodegen generate`).
   - The bundle ids are `com.wordtaker.app` and `com.wordtaker.app.keyboard`;
     change the prefix to one your team owns if needed.
   - The **App Group** `group.com.wordtaker.shared` must exist in your developer
     account and be enabled on both targets (it is already in both
     `.entitlements` files). Update the id in `RelayConfig.appGroupIdentifier`
     and both entitlements if you use a different group.
4. **Enable the keyboard on device:** Settings → General → Keyboard → Keyboards →
   Add New Keyboard → WordTaker Keyboard → then enable **Allow Full Access**.
5. **Run tests:** select the `WordTaker` scheme → Product → Test
   (`WordTakerTests`).

## What requires full Xcode / a device / signing

- Compiling, running, and producing an `.ipa` (full Xcode + `xcodebuild`).
- Microphone + on-device speech recognition (best on a real device; permission
  prompts and on-device ASR availability vary by simulator).
- App Group data sharing across the app and keyboard (works on device; needs the
  App Group provisioned under your team).
- Code signing / provisioning profiles for any device install or TestFlight.
